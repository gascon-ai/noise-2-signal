Lesson 10.1video

Define the roles

Outcome: one named agent per department, each with a written job description, before you build anything.

The departments are standing. Content attracts, Outreach prospects, Sales closes, Admin runs the day-to-day. Until now you have run each one by hand. The shift in this module is simple: every department gets a hire. Not a vague helper, a named agent with a real job.

So you do what a founder does before posting a role. You write the job description. A content lead who owns the Content engine. An outreach rep who owns the Instantly and GHL campaigns. A sales ops manager who keeps the pipeline clean in your CRM. Name them, give each one a mandate, a scope, and a hard line on what they never do without you.

What each role holds

  • The name. Give the agent an identity so you can hand it work by name.
  • The department. The one bucket it owns, and nothing outside it.
  • The mandate. The outcome it is responsible for, in one line.
  • The guardrails. What it does on its own, and what it must bring to you first.

Do this

  1. List your departments. One agent per department, no overlap.
  2. Write the job description for each, mandate and guardrails included.
  3. Keep them short. A role is a paragraph, not a manual.

Done when: every department has a named agent with a one-line mandate and a clear line it does not cross.

Key takeaway: you cannot delegate a role you have not defined. Name the hire and write its job before you build it.

Build the agents

Outcome: each role becomes a real agent, a skill plus memory plus tools, pointed at its department's brief.

A job description is not an employee. To make the agent real you give it three things, the same three you have been building all course. A skill, the repeatable workflow it runs. Memory, its department's brief and memory folder from Module 2. Tools, the connectors it actually touches: Instantly for the outreach rep, the CRM for sales ops, Supabase and the scraper for the content lead.

Point each agent at its room and it knows that room cold. The content lead reads the Content brief and runs the eight-seat engine. The outreach rep reads the Outreach brief and works the campaigns. Nothing generic. Each agent is the department brief plus the hands to act on it.

Prompt: Build me a content-lead agent. Read the Content department brief and memory folder as its context. Give it the content skills it needs and access to Supabase and the scraper. Write it as a Claude Code subagent scoped to the content bucket only.

Do this

  1. For each role, define it as a subagent in Claude Code.
  2. Point it at its department brief and memory folder so it loads the right context.
  3. Grant only the tools that department uses. No more.

Key takeaway: an agent is a skill, a memory, and a set of tools aimed at one room. Build it once, it knows that room every time.

Lesson 10.3video

Delegate

Outcome: you hand a job to the right agent the way you would hand it to a hire, with the outcome and the guardrails up front.

Here is where the work leaves your hands. You do not do the task. You assign it. The skill is the same one you use with a real employee: be clear about the outcome, hand over the context, set the guardrails, then let them run.

A weak hand-off is "write some posts." A strong one names the outcome, the constraints, and the line for when to check back. The agent already carries the department context, so you are not re-explaining the business. You are giving a brief.

Prompt: Content lead, draft this week's five posts from the pipeline. Voice and banned words per the soul file. No claim without a receipt. Hold anything that needs a stat I have not confirmed, and bring me the drafts before anything ships.

Do this

  1. Pick a real job and hand it to the agent that owns that department.
  2. State the outcome, the constraints, and where it must stop and check with you.
  3. Let it work. Resist doing the task yourself.

Done when: you assign a real job to an agent, give it the outcome and the guardrails, and it does the work.

Key takeaway: delegate like a manager, not a doer. Outcome plus guardrails, then hands off the keyboard.

Review and improve

Outcome: you become the manager. You check the output, give feedback, and the feedback gets written back into the agent's memory so it improves.

Delegation without review is just hope. The job is not done when the agent finishes, it is done when you have checked the work. So you put on the manager hat: read what came back, approve it or send it back, and say exactly what was off.

The part that makes this compound: the feedback does not die in the chat. You write it back into the agent's memory folder, the same folder from Module 2. "Always lead with the receipt." "Never use that closer." Next time, the agent already knows. The correction sticks, so you give it once, not every week.

Do that across every department and you have an org chart that runs itself. Agents do the work, you review and steer, and each agent gets sharper from your notes. You went from doing the business to managing it.

Do this

  1. Review the agent's output. Approve it or send it back with a specific note.
  2. Write the lesson into the agent's memory or brief, not just the chat.
  3. Watch the same mistake not come back. That is the loop closing.

Key takeaway: review is where the agent learns. Feedback in the chat is lost. Feedback in memory makes it improve.

What you walk away with

A team. Every department has a named agent that knows its room, the tools to act on it, and a memory that gets sharper every time you correct it. You hand out the work and review what comes back, the way a founder runs hires instead of doing every job. The business runs, and you manage. The one thing left is the guardrail around all of it: keeping the whole machine safe and honest.
