Backups: the work backs itself up

Outcome: every session syncs your whole setup, skills and memory included, to a private GitHub repo, so you can never lose the brain you built.

You have spent ten modules furnishing a second brain. Skills, memory files, briefs, the soul file. Now imagine the drive dies. If that work lives in one place, one bad day erases it. So you make the backup automatic and you stop thinking about it.

A hook does it. When a session ends, the hook commits your whole workspace, skills, memory, briefs and all, and pushes it to a private GitHub repo. No button to remember, no discipline to keep. The work backs itself up.

The receipt is this exact system. It runs on the backup hook. It once drifted, the local copy and the repo fell out of sync, and the fix was to harden the hook so it can never drift again. That is the difference between a backup you hope is running and one you have proven.

Do this

  1. Create a private GitHub repo for your workspace.
  2. Install the auto-backup hook so it commits and pushes at the end of every session.
  3. Force a break. Change a file, end a session, and confirm the commit landed on GitHub.

Done when: you finish a session and a fresh commit shows up in your private repo without you touching git.

Key takeaway: a backup you have to remember is not a backup. The hook makes it automatic, and the second brain survives a dead drive.

Data safety: the rules that prevent a disaster

Outcome: a short set of standing rules that stop the AI from ever destroying a file, baked into your soul file so every session obeys them.

The AI is fast and it is willing. Tell it to clean up a folder and it will, including the parts you wanted kept. On a cloud drive, Dropbox or Google Drive or iCloud, a careless move or delete can trash the original and leave a zero-byte ghost in its place. Speed without rules is how you lose a year of work in one command.

The rule that prevents most of it

Never move or delete files straight on a cloud drive. Always copy, verify the copy is real and larger than zero, then delete the original. Copy, check, then remove. In that order, every time.

  • Copy first. Get the file to its new home before anything happens to the old one.
  • Verify the copy. Confirm it exists and has real size at the destination. No verify, no delete.
  • Then delete the original. Only after the copy is proven. Through the cloud provider, never a raw delete on a synced mount.

These are not suggestions you hope it remembers. They go in the soul file as hard rules, so every session inherits them, the same way Foundation taught it where things live.

Do this

  1. Add the data-safety rules to your soul file as standing instructions.
  2. Mark your cloud-synced folders by name so the AI knows where the rules bite hardest.
  3. Test it. Ask it to reorganize a throwaway folder and watch it copy, verify, then delete.
Download · the data-safety rules
Done when: the AI refuses to move or delete on a cloud drive and copies, verifies, then removes instead.

Key takeaway: copy, verify, then delete. One rule, written once into the soul file, is the seatbelt that prevents the disaster you would not survive.

The fact-gate

Outcome: no claim, stat, or quote leaves the building unverified, across every department, not just content.

The genius with amnesia is also a confident guesser. It will invent a statistic that sounds right, attribute a quote to the wrong person, and state it all with a straight face. In content that costs you credibility. In a proposal or a client report it costs you the client.

You met the fact-gate in the content engine, the check seat that holds a draft against its sources before it ships. Here you promote it from a content step to a standing rule for the whole business. Every number, every quote, every "studies show" gets gated.

Rule: Before you state any fact, statistic, quote, or claim, verify it against a real source and cite it. If you cannot verify it, say so plainly and do not assert it. No claim ships unverified.

Put it in the soul file so it is always on, and lean on the tools you already have. Real sources beat a confident guess every time: a web search for a stat, the meeting transcript in Fathom for what a client actually said, your own archive for what you actually decided.

Do this

  1. Add the fact-gate rule to your soul file as a business-wide standing instruction.
  2. Tell it which sources count: live web for stats, Fathom for quotes, your archive for your own history.
  3. Make it label anything it cannot verify, instead of asserting it.
Download · the fact-gate
Done when: the system flags an unverifiable claim and asks rather than inventing one.

Key takeaway: no receipt, no claim. The same gate that protects your content now protects every word the business sends.

The recovery runbook

Outcome: a one-page playbook that restores the whole system from scratch, so a broken machine is a minor inconvenience, not a crisis.

Backups only matter if you can come back from them. Most people have a backup somewhere and no idea how to restore it under pressure. So you write the playbook once, while nothing is on fire, and a dead laptop becomes a one-hour problem.

It is short on purpose. When something breaks you do not want to think, you want to follow steps.

  • Clone fresh from GitHub. Your whole brain lives in the private repo from lesson 11.1. Pull it down to the new machine.
  • Re-auth the tools. Reconnect Claude Code and your MCP tools, Notion, Slack, Stripe, Fathom, the connectors each department runs on.
  • Confirm and continue. Open a bucket, check the brief loads and the skills run, and you are back to work in minutes.

Keep the runbook in your admin bucket and treat it like a fire drill. The first time you follow it should not be the day the drive dies.

Do this

  1. Write the runbook: clone from GitHub, re-auth the tools, verify, in plain steps.
  2. List every tool you would need to reconnect, so nothing is missing under pressure.
  3. Run the drill once on a spare machine or a clean folder and time it.

Key takeaway: a backup you cannot restore is a souvenir. The runbook turns the worst day into a short one.
